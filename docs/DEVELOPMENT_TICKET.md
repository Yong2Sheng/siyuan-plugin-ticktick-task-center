# Ver. 2思源插件开发 Ticket：滴答清单任务卡片与任务中心

思源插件开发 Ticket：滴答清单任务卡片与任务中心

## 1. 项目概述

我希望在思源笔记中，为具体文档关联一个或多个滴答清单任务，并通过醒目的任务卡片展示任务名称、跳转链接和当前状态。

同时，插件需要提供一个独立的“任务中心”视图，动态汇总整个思源工作空间中的全部任务卡片。

任务中心不是第二份需要单独维护的任务数据库，而是对各文档中任务卡片数据的动态查询、分类、统计和操作界面。

整体数据关系如下：

```text
具体文档中的任务卡片
        ↓
任务块的结构化属性（唯一数据源）
        ↓
任务中心动态查询、分类和汇总
```

当原文档中的任务名称、链接或状态发生变化时，任务中心中的对应记录应随之更新。

从任务中心修改任务时，也应直接修改原文档中的同一个任务块，而不是修改另一份独立记录。

插件第一版不接入 TickTick API，不处理登录、OAuth 或双向同步。核心目标是：

```text
保存 TickTick 任务链接
快速打开对应任务
在思源中维护可见任务状态
避免手动编辑 HTML 和 CSS
以结构化数据支持全局任务汇总
通过任务中心集中查看和管理任务
```

---

## 2. 背景与现有痛点

目前，我已经通过 HTML 块实现了可点击的任务卡片：

- 卡片位于笔记开头；
- 显示“滴答清单任务”身份标签；
- 显示任务名称和任务状态；
- 点击任务按钮可以打开滴答清单中的对应任务；
- 卡片样式可以跟随思源主题主色。

当前方案在展示效果上已经基本满意，但维护成本较高：

- 新建笔记时需要复制 HTML 模板；
- 需要手动粘贴 TickTick 任务链接；
- 需要手动填写任务名称；
- 状态变化时，需要修改 Emoji、状态文字、颜色和 CSS；
- 很容易误删 HTML 标签、属性或样式；
- 任务信息不是结构化数据；
- 无法可靠汇总不同文档中的任务；
- 无法一键查看所有“进行中”“等待回复”或“已阻塞”的任务；
- 如果使用全文搜索，可能搜索到普通笔记中的同名文字；
- 如果使用 Tag 保存状态，会产生重复数据和同步风险。

因此，需要将现有任务卡片从手写 HTML 转换为插件管理的结构化组件，并增加动态“任务中心”。

## 3. 设计原则

### 3.1 单一数据源

每张任务卡片对应一个思源块。

任务标题、TickTick 链接、状态和时间信息均保存在该块的结构化属性中。任务中心不保存这些字段的独立副本，只动态读取原任务块。

必须避免出现：

```text
原文档任务卡片：▶️ 进行中
任务中心记录：⏳ 等待回复
```

任务状态的唯一数据来源应为：

```text
custom-ticktick-status
```

任务名称、链接和时间信息同样直接读取原任务块属性。

---

### 3.2 显示与数据分离

状态 ID 是稳定的数据值，例如：

```text
in-progress
```

显示层可以将其渲染为：

```text
▶️ 进行中
```

以后即使更换 Emoji、中文名称、颜色或布局，也不应影响任务查询和数据迁移。

因此，任务中心不应通过搜索：

```text
▶️ 进行中
```

来识别状态，而应查询：

```text
custom-ticktick-status = in-progress
```

---

### 3.3 任务中心是动态视图，不是第二份数据库

任务中心可以呈现为表格、列表或卡片式界面，但其本质是对任务块的动态查询结果。

任务中心中显示的以下信息：

```text
任务名称
任务状态
TickTick 链接
来源文档
来源笔记本
创建时间
更新时间
```

均来自原任务块或其所在文档上下文，不应再维护一份独立数据。

即使未来任务中心采用类似思源数据库、属性视图或看板的显示形式，也必须遵循：

```text
原任务块属性 = 唯一数据源
任务中心界面 = 动态视图
```

---

### 3.4 状态不以 Tag 或全文搜索为主索引

全文搜索和 Tag 可以作为辅助能力，但不作为任务状态的正式数据基础。

原因包括：

- “进行中”等文字可能出现在普通笔记正文中；
- Emoji 可能存在不同 Unicode 变体；
- 插件渲染内容不一定等于底层全文索引内容；
- 状态名称以后可能修改；
- Tag 更适合主题、项目和知识分类；
- 状态是互斥且频繁变化的结构化字段；
- 状态 Tag 与块属性之间可能发生不一致。

正式查询应基于结构化状态 ID。

---

### 3.5 主题适配

Emoji 本身由系统字体渲染，颜色通常固定。

状态颜色不应用于强行修改 Emoji，而应应用于：

```text
状态文字
状态徽章边框
状态徽章背景
卡片边框
卡片低透明度背景
选中状态和筛选按钮
```

卡片需要在浅色和深色主题中保持可读。

## 5. 核心使用场景

### 5.1 为当前文档插入任务卡片

例如，我新建一篇思源笔记：

```text
Add collection and group plotting method
```

在编辑器中执行：

```text
/滴答任务卡片
```

随后弹出表单：

```text
任务名称：
[Add collection and group plotting method]

滴答任务链接：
[粘贴 TickTick URL]

任务状态：
[▶️ 进行中 ▼]

[插入任务卡片]
```

要求：

- 任务名称默认读取当前文档标题；
- 用户可以修改默认名称；
- TickTick URL 必填；
- 默认状态为 `▶️ 进行中`；
- 默认插入位置为当前文档顶部；
- 当前文档允许插入多个任务卡片。

---

### 5.2 快速切换任务状态

例如任务从：

```text
▶️ 进行中
```

变为：

```text
⏳ 等待回复
```

用户可以：

```text
点击状态徽章
→ 选择“⏳ 等待回复”
```

或者：

```text
右键任务卡片
→ 更新任务状态
→ ⏳ 等待回复
```

更新后，插件应自动修改：

```text
状态 ID
状态 Emoji
状态文字
状态徽章颜色
状态徽章背景
卡片强调色
updatedAt
```

如果任务中心已经打开，任务中心中的分类、计数和列表也应刷新。

---

### 5.3 快速打开 TickTick 任务

任务卡片主按钮显示：

```text
打开任务：<任务名称> ↗️
```

点击后：

```text
在浏览器或滴答清单客户端中打开对应 URL
```

插件只负责保存和打开 URL，不负责验证任务是否存在，也不负责读取 TickTick 中的真实状态。

---

### 5.4 编辑任务信息

右键任务卡片后提供：

```text
编辑任务信息
```

允许修改：

```text
任务名称
TickTick 链接
任务状态
```

用户不需要接触 HTML、CSS 或块属性。

---

### 5.5 删除任务卡片

右键任务卡片后提供：

```text
删除任务卡片
```

删除前进行确认：

```text
确定要删除这个任务卡片吗？

这里只会删除思源中的任务卡片，
不会删除 TickTick 中的实际任务。

[取消] [删除]
```

---

### 5.6 在任务中心集中查看任务

插件提供独立的“任务中心”界面。

默认显示所有非终态任务：

```text
⚪ 待开始
▶️ 进行中
⏳ 等待回复
⛔ 已阻塞
```

顶部显示汇总：

```text
全部：48
非终态：12
终态：36

⚪ 待开始：3
▶️ 进行中：5
⏳ 等待回复：2
⛔ 已阻塞：2
✅ 已完成：30
❌ 已失败：2
⏹️ 已取消：4
```

点击某个状态后，只显示该状态的任务。

另外我还想要以列表的形式直观地显示所有的任务：

|任务|状态|TickTick 链接|所属文档|更新时间|
| ------------------------| ----------| ---------------| ----------| ------------|
|DS9 Adaptor|进行中|URL|Photozpy|2026-07-11|
|Eliza GRB localization|等待回复|URL|COSI|2026-07-10|

点击任务中的名称，可以直接跳转到对应的思源笔记文档。

---

### 5.7 从任务中心修改原任务

在任务中心中将：

```text
▶️ 进行中
```

修改为：

```text
✅ 已完成
```

插件应直接修改原任务块的：

```text
custom-ticktick-status = completed
```

修改后：

```text
原文档卡片更新为 ✅ 已完成
任务中心中的任务移动到“已完成”
非终态和终态数量同步变化
updatedAt 自动更新
```

任务中心和文档任务卡片操作的是同一个任务对象。

---

### 5.8 区分不同结束结果

任务结束后，需要区分：

```text
✅ 已完成
已达到预期目标。

❌ 已失败
任务已经尝试执行，但最终未达到预期目标。

⏹️ 已取消
任务被主动停止，不再继续，但不应被视为失败。
```

同时：

```text
⛔ 已阻塞
```

不是终态。阻塞原因解决后，任务仍可以恢复为：

```text
▶️ 进行中
```

## 6. 任务状态系统

固定支持以下七种状态：

|状态 ID|显示文字|图标|状态类型|含义|建议强调色|颜色说明|
| ---------| ----------| :----: | ----------| ------------------------------| ---------------------| ----------------------------------------|
|`todo`|待开始|⚪|非终态|已创建，但尚未启动|中性灰 / 次要文字色|弱化显示，表示尚未进入工作流程|
|`in-progress`|进行中|▶️|非终态|当前正在推进|主题主色|当前最活跃的状态，应具有较高视觉优先级|
|`waiting`|等待回复|⏳|非终态|等待他人反馈、审批或外部输入|黄色 / 琥珀色|属于正常等待，不代表任务出现故障|
|`blocked`|已阻塞|⛔|非终态|暂时无法继续，但未来可能恢复|橙色 / 警告色|表示存在需要处理的依赖或问题|
|`completed`|已完成|✅|终态|已达到预期目标|绿色 / 成功色|表示任务成功结束|
|`failed`|已失败|❌|终态|已终止，且未达到预期目标|红色 / 错误色|表示任务尝试过，但最终失败|
|`cancelled`|已取消|⏹️|终态|主动停止，不再继续执行|灰色 / 禁用文字色|不属于成功或失败，不使用红色强调|

### 6.1 非终态

```text
⚪ 待开始
▶️ 进行中
⏳ 等待回复
⛔ 已阻塞
```

这些状态表示任务仍有可能继续推进。

---

### 6.2 终态

```text
✅ 已完成
❌ 已失败
⏹️ 已取消
```

这些状态表示任务当前已经结束。

第一版不需要限制终态重新切换为非终态，但界面和数据结构必须明确区分两类状态。

---

### 6.3 推荐状态转换

正常流程：

```text
⚪ 待开始
→ ▶️ 进行中
→ ✅ 已完成
```

等待流程：

```text
▶️ 进行中
→ ⏳ 等待回复
→ ▶️ 进行中
→ ✅ 已完成
```

阻塞流程：

```text
▶️ 进行中
→ ⛔ 已阻塞
→ ▶️ 进行中
→ ✅ 已完成
```

失败流程：

```text
▶️ 进行中
→ ❌ 已失败
```

取消流程：

```text
⚪ 待开始
→ ⏹️ 已取消
```

## 8. 任务卡片显示设计

### 8.1 总体布局

理想单行布局：

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ ✅ 滴答清单任务  [打开任务：任务标题 ↗️]                [▶️ 进行中]      │
└─────────────────────────────────────────────────────────────────────────┘
```

布局要求：

- 左侧：固定身份标签；
- 中间：TickTick 主按钮；
- 右侧：状态徽章；
- 状态徽章固定右对齐；
- 宽度不足时允许换行；
- 长标题不能无限撑宽卡片；
- 浅色和深色主题均可读。

推荐布局：

```css
.ticktick-task-card {
  display: flex;
  align-items: center;
  gap: 10px;
  flex-wrap: wrap;
}

.ticktick-task-card__status {
  margin-left: auto;
  flex: 0 0 auto;
}
```

---

### 8.2 卡片基础样式

建议保留现有设计语言：

```text
外边距：12px 0 18px
内边距：12px 16px
边框：1.5px
圆角：10px
字体大小：15px
背景：强调色的低透明度背景
```

基础变量优先使用思源主题变量：

```css
var(--b3-theme-primary)
var(--b3-theme-on-primary)
```

状态色应通过统一 CSS 变量集中管理，避免重复内联样式：

```css
.ticktick-task-card {
  --task-status-color: var(--b3-theme-primary);
  --task-status-background: color-mix(
    in srgb,
    var(--task-status-color) 10%,
    transparent
  );
}
```

不同状态通过属性选择器切换：

```css
.ticktick-task-card[data-status="in-progress"] {
  --task-status-color: var(--b3-theme-primary);
}
```

具体成功色、警告色和错误色应优先使用思源主题语义变量；若主题没有对应变量，再使用插件后备色。

---

### 8.3 身份标签

固定显示：

```text
✅ 滴答清单任务
```

说明：

- 这里的 `✅` 表示这是一个任务组件；
- 不代表任务已经完成；
- 真正状态由右侧状态徽章表达；
- 后续可以在设置中允许用户隐藏身份标签或替换身份 Emoji。

样式：

```text
主题主色
字重 700
不换行
不可点击
```

---

### 8.4 主按钮

显示：

```text
打开任务：<任务名称> ↗️
```

建议样式：

```text
背景：主题主色
文字：主题反色
圆角：6px
字重：700
Hover：轻微改变亮度或透明度
```

要求：

- 一眼可辨识为可点击按钮；
- 任务标题过长时可以截断或换行；
- Hover 时可显示完整标题；
- 不能挤压或覆盖状态徽章。

可使用：

```css
min-width: 0;
overflow-wrap: anywhere;
```

若采用单行截断：

```css
overflow: hidden;
text-overflow: ellipsis;
white-space: nowrap;
```

---

### 8.5 状态徽章

统一规则：

```text
Emoji + 状态文字
圆角胶囊标签
状态色文字
状态色边框
低透明度状态色背景
字重 700
不换行
固定右对齐
```

示例：

```text
⚪ 待开始
▶️ 进行中
⏳ 等待回复
⛔ 已阻塞
✅ 已完成
❌ 已失败
⏹️ 已取消
```

---

### 8.6 各状态视觉语义

#### ⚪ 待开始

```text
中性灰色文字
中性灰色边框
浅灰背景
```

任务已创建，但尚未正式启动。

---

#### ▶️ 进行中

```text
思源主题主色
主题色边框
主题色低透明度背景
```

任务当前正在被主动推进，视觉优先级最高。

---

#### ⏳ 等待回复

```text
黄色或琥珀色
浅黄色背景
黄色或琥珀色边框
```

适用于：

```text
等待合作者回复
等待导师反馈
等待观测批准
等待审稿意见
等待 telescope response
等待外部数据发送
```

这是正常等待，不应使用红色。

---

#### ⛔ 已阻塞

```text
橙色或警告色
浅橙背景
橙色边框
```

适用于：

```text
代码依赖尚未解决
关键外部数据缺失
服务器问题
软件环境损坏
观测条件不满足
权限或审批问题
```

阻塞是非终态，问题解决后可以恢复。

---

#### ✅ 已完成

```text
绿色或成功色
浅绿色背景
绿色边框
```

任务达到预期目标并成功结束。

可选增强：

```text
整个卡片略微降低视觉权重
主按钮文字由“打开任务”改为“查看任务”
```

第一版可以只改变徽章和卡片强调色。

---

#### ❌ 已失败

```text
红色或错误色
浅红背景
红色边框
```

适用于：

```text
实验最终没有成功
技术方案验证失败
申请或提案未获批准
任务经过尝试后确认不可行
```

表示任务已经结束且未达到目标。

---

#### ⏹️ 已取消

```text
灰色或禁用文字色
浅灰背景
灰色边框
整体视觉权重降低
```

适用于：

```text
需求已经不再需要
计划发生变化
任务被另一个方案取代
重复任务被关闭
用户主动决定不再继续
```

取消不是失败，不使用红色强调。

## 10. 存储与渲染方案

### 10.1 方案 A：HTML 块加 `data-*` 属性

```html
<div
  class="ticktick-task-card"
  data-version="1"
  data-title="Add collection and group plotting method"
  data-url="https://dida365.com/webapp/#p/..."
  data-status="in-progress"
>
  ...
</div>
```

优点：

- 与现有 HTML 模板兼容；
- 容易迁移旧卡片；
- 导出 HTML 时可以保留基本展示。

缺点：

- DOM 生命周期和交互更新更复杂；
- 用户可能手动破坏 HTML；
- 全局查询需要解析 HTML；
- 内联样式不利于维护。

---

### 10.2 方案 B：普通块加块属性加插件渲染

```text
块内容：
滴答任务卡片

块属性：
custom-ticktick-card
custom-ticktick-version
custom-ticktick-title
custom-ticktick-url
custom-ticktick-status
custom-ticktick-created-at
custom-ticktick-updated-at
```

插件根据块属性渲染为卡片。

优点：

- 数据结构清晰；
- 状态查询可靠；
- 适合任务中心动态汇总；
- 不依赖用户编辑 HTML；
- 样式可以集中维护；
- 更适合长期扩展。

缺点：

- 插件未加载时需要降级显示；
- 导出 Markdown 或 HTML 时需要额外策略；
- 需要处理块渲染、刷新和编辑状态。

---

### 10.3 推荐方案

```text
方案 B 为主
方案 A 用于旧模板迁移和导出降级
```

具体策略：

- 思源内部使用块属性保存结构化任务数据；
- 插件负责渲染任务卡片；
- 任务中心通过块属性查询任务；
- 插件未加载时，普通块仍保留可识别的任务标题或链接；
- 导出时可生成标准链接文本或标准 HTML；
- 旧 HTML 卡片可通过迁移工具转换。

## 11. 任务中心

### 11.1 定位

任务中心是插件的核心全局视图，用于动态汇总、筛选和操作所有任务卡片。

任务中心不是独立数据源，也不要求用户维护第二份表格。

---

### 11.2 入口

建议提供：

```text
插件顶部工具栏按钮
右侧 Dock 面板
命令面板：打开任务中心
```

可选增加：

```text
/任务中心
```

显示形态建议：

- 右侧 Dock：适合快速查看；
- 独立页签或弹窗：适合浏览大量任务；
- MVP 可以先实现一种形态；
- 后续可同时支持 Dock 和独立页签。

---

### 11.3 默认视图

默认显示：

```text
所有非终态任务
```

顶部提供三个一级分类：

```text
全部
非终态
终态
```

并提供七种状态筛选：

```text
⚪ 待开始
▶️ 进行中
⏳ 等待回复
⛔ 已阻塞
✅ 已完成
❌ 已失败
⏹️ 已取消
```

建议显示状态数量：

```text
▶️ 进行中 5
⏳ 等待回复 2
⛔ 已阻塞 1
```

---

### 11.4 任务中心列表内容

每条任务记录至少显示：

```text
任务状态 Emoji 和状态文字
任务名称
所属文档
所属笔记本
最后更新时间
```

示意：

```text
▶️ Add collection and group plotting method

来源：
Photozpy / Image Collection Refactoring

更新于：
2026-07-11 15:20

[定位卡片] [打开 TickTick]
```

---

### 11.5 任务中心操作

每条任务应支持：

```text
定位到原文档中的任务卡片
打开 TickTick 任务
修改任务状态
编辑任务信息
复制 TickTick 链接
```

可选支持：

```text
打开来源文档
删除任务卡片
```

删除操作必须明确说明只删除思源卡片，不删除 TickTick 任务。

---

### 11.6 从任务中心修改状态

例如：

```text
▶️ 进行中
→ ✅ 已完成
```

插件直接修改原任务块：

```text
custom-ticktick-status = completed
custom-ticktick-updated-at = <当前时间>
```

随后：

```text
原文档卡片重新渲染
任务从“进行中”列表移出
任务进入“已完成”列表
状态统计自动更新
```

---

### 11.7 动态刷新策略

任务中心应在以下情况刷新：

```text
插入新任务卡片
修改任务名称
修改 TickTick 链接
修改任务状态
删除任务卡片
从任务中心修改任务
打开任务中心
```

MVP 刷新方式：

```text
打开任务中心时重新查询全部任务
插件自身修改任务后立即刷新
提供手动刷新按钮
```

后续版本：

```text
监听思源块更新事件
在原文档变化时自动刷新任务中心
实现近实时动态更新
```

---

### 11.8 搜索、筛选和排序

虽然任务中心已经避免了日常使用思源全文搜索，但仍应提供辅助能力。

MVP 建议支持：

```text
按状态筛选
按任务名称搜索
按更新时间排序
```

后续支持：

```text
按所属文档筛选
按所属笔记本筛选
按创建时间排序
按任务名称排序
按任务状态排序
保存筛选条件
```

默认排序：

```text
最后更新时间：从新到旧
```

---

### 11.9 空状态与错误状态

没有匹配任务时显示：

```text
当前没有“▶️ 进行中”的任务。
```

查询失败时显示：

```text
任务列表加载失败。

[重试]
```

单条任务的来源块已被删除时：

```text
该任务卡片已不存在。

[从列表移除] [刷新]
```

正常情况下，任务中心不应保留被删除任务的长期缓存。

---

### 11.10 性能策略

MVP 可以在打开任务中心时查询所有带有：

```text
custom-ticktick-card = true
```

的任务块。

如果任务数量增加，后续可以考虑：

```text
分页加载
虚拟列表
按需读取来源文档信息
短期内存缓存
增量刷新
```

缓存只能作为性能优化，不能成为新的任务数据源。

## 12. 与思源原生索引方式的关系

### 12.1 全文搜索

如果底层块内容中保留状态文字，用户理论上可以搜索：

```text
▶️ 进行中
```

但全文搜索仅作为备用方式，不作为主索引。

原因：

- 可能命中普通笔记正文；
- 显示文字变化会影响搜索；
- Emoji 编码可能不同；
- 插件渲染文字不一定进入全文索引；
- 无法保证只返回任务卡片。

---

### 12.2 Tag

默认不将任务状态保存为 Tag。

原因：

- Tag 更适合主题和知识分类；
- 状态频繁变化；
- 状态互斥；
- 同步旧 Tag 和新 Tag 容易出错；
- 会污染普通 Tag 系统。

后续可提供可选设置：

```text
将任务状态同步为思源 Tag
```

但 Tag 只能作为辅助入口，不能成为状态唯一数据源。

---

### 12.3 块属性

块属性是任务状态和任务信息的正式存储方式。

任务中心通过结构化块属性完成：

```text
识别任务块
按状态分类
统计任务数量
读取任务标题和链接
更新原任务状态
```

---

### 12.4 思源数据库或属性视图

未来可以将任务中心增加为表格或看板式视觉模式，但不应因此创建第二份独立任务记录。

即使采用类似数据库的界面，也必须遵守：

```text
原任务块属性 = 唯一数据源
任务中心表格或看板 = 动态视图
```

## 17. MVP 验收标准

### 17.1 任务卡片

- 可以通过 `/滴答任务卡片` 插入新卡片；
- 插入时可以填写任务名称、TickTick URL 和状态；
- 任务名称默认读取当前文档标题；
- 默认状态为 `▶️ 进行中`；
- 默认插入位置为当前文档顶部；
- 当前文档允许多个任务卡片；
- 卡片显示 `✅ 滴答清单任务` 身份标签；
- 主按钮显示 `打开任务：<任务名称> ↗️`；
- 点击主按钮可以打开 TickTick URL；
- 状态徽章固定在右侧；
- 宽度不足时布局能够合理换行；
- 点击状态徽章可以切换七种状态；
- 七种状态使用统一 Emoji、状态文字和强调色；
- 可以明确区分非终态和终态；
- 可以明确区分 `⛔ 已阻塞`​、`❌ 已失败`​ 和 `⏹️ 已取消`；
- 右键菜单可以编辑任务名称、链接和状态；
- 右键菜单可以复制 TickTick 链接；
- 右键菜单可以删除任务卡片；
- 删除卡片不会删除 TickTick 中的任务；
- 用户不需要编辑 HTML、CSS 或块属性；
- 浅色和深色主题下均可正常阅读。

---

### 17.2 任务中心

- 提供至少一个打开任务中心的入口；
- 可以动态查询全部任务卡片；
- 默认显示所有非终态任务；
- 可以切换“全部”“非终态”“终态”；
- 可以按七种状态筛选；
- 显示每种状态的任务数量；
- 每条任务显示状态、名称、来源文档和更新时间；
- 可以定位到原文档中的任务卡片；
- 可以打开对应 TickTick 任务；
- 可以从任务中心修改任务状态；
- 从任务中心修改状态后，原文档卡片同步更新；
- 从原文档修改状态后，重新打开或刷新任务中心可以看到新状态；
- 提供手动刷新按钮；
- 插件修改任务后立即刷新任务中心；
- 删除任务卡片后，任务中心不再显示该任务；
- 任务中心不保存任务状态独立副本。

---

### 17.3 数据

- 任务数据以结构化块属性保存；
- 状态查询基于状态 ID，而不是显示文字或 Emoji；
- `custom-ticktick-status` 是状态唯一数据源；
- 任务中心是动态视图；
- Tag 和全文搜索不作为主索引；
- 不接入 TickTick API；
- 不要求登录、授权或同步。

## 20. 开发实现建议

建议从思源官方插件示例或常用的 Vite/Svelte 插件模板起步。

具体 API、模板版本和最低思源版本应在正式开发时，根据实际开发环境再次确认，不在 Ticket 中硬编码为“当前最新版本”。

代码结构建议至少分为：

```text
task-card/
  数据读取与写入
  卡片渲染
  交互菜单
  编辑表单

task-center/
  任务查询
  状态统计
  列表渲染
  筛选和排序
  定位与打开操作

status/
  状态类型
  状态配置
  状态颜色
  状态校验

migration/
  旧 HTML 识别
  状态映射
  数据转换

settings/
  默认状态
  默认插入位置
  任务中心默认视图
```

避免把任务查询、卡片 DOM 和任务中心逻辑全部写在同一个模块中。

## 21. 一句话定义

一个用于在思源笔记中关联 TickTick 任务的轻量插件：通过结构化任务卡片展示任务链接和七种任务状态，并通过动态“任务中心”集中汇总、筛选和管理全部任务，而无需手动维护 HTML、CSS、Tag 或独立任务数据库。
